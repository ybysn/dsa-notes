# 学习计划生成工作流

这个工作流用于把一本书、一组电子书、课程资料或文档，整理成可执行的系统学习计划。

适用场景：

- 学习数据结构与算法
- 学习 Python、Java、Go、前端、后端、数据库等技术
- 学习数学、英语、考试科目或专业课程
- 把一本书拆成每日学习任务

## 1. 准备资料

把资料放到一个独立文件夹中，例如：

```text
topic-learning/
├── book.pdf
├── course-notes.pdf
└── references/
```

建议只放与当前主题有关的资料，避免计划范围过大。

如果资料很多，先确定主教材：

```text
主教材：必须完整跟读
辅助资料：用于查漏补缺
练习资料：用于训练和验收
```

## 2. 提取资料结构

先读取资料目录，确认章节结构。

需要得到：

- 书名或课程名
- 总页数或总课时
- 章节目录
- 每章主题
- 是否包含练习、项目、案例、附录

输出一份简要结构，例如：

```text
第 1 章：基础知识
第 2 章：核心概念
第 3 章：实践应用
第 4 章：进阶主题
```

## 3. 制定总学习计划

先做一份全局计划，回答：

- 学多久
- 每周学几天
- 每天学多久
- 分几个阶段
- 每个阶段目标是什么
- 最终验收标准是什么

推荐文件名：

```text
主题学习计划.md
```

推荐结构：

```text
# 主题学习计划

## 学习节奏
## 阶段 1：基础
## 阶段 2：核心
## 阶段 3：进阶
## 阶段 4：实践
## 每周固定产出
## 最终验收标准
```

原则：

- **总计划只管方向，不写太细**
- **阶段目标要清楚**
- **每个阶段都要有可验收结果**

## 4. 制定每周计划

在总计划基础上，把学习内容拆成周。

推荐文件名：

```text
主题每周学习计划.md
```

每周包含：

- 对应章节
- 本周目标
- 周一到周日安排
- 代码任务或实践任务
- 练习重点
- 验收标准

推荐结构：

```text
## 第 1 周：主题名称

对应章节：

本周目标：

每日安排：
- 周一：
- 周二：
- 周三：
- 周四：
- 周五：
- 周六：
- 周日：

实践任务：

练习重点：

验收标准：
```

原则：

- **每周最好围绕一个明确主题**
- **每天任务不要过载**
- **周末留给补缺、复盘、重写或总结**

## 5. 先做一天样张

不要一开始就批量生成所有每日计划。

先做第 1 天样张，让格式稳定下来。

推荐文件名：

```text
day01_主题.md
```

每日文件建议包含：

```text
# Day 01：标题

对应周次：
对应章节：
预计用时：

## 今日目标
## 阅读任务
## 核心讲解
## 今日实践任务
## 今日练习题
## 复杂度/原理/方法分析
## 今日验收
## 今日复盘模板
## 明天预告
```

如果是编程类主题，使用：

```text
## 今日代码任务
## 复杂度分析
```

如果是非编程类主题，替换为：

```text
## 今日练习任务
## 方法总结
```

原则：

- **每日文件必须能独立使用**
- **阅读任务后面要有改写讲义**
- **重点概念用加粗标出**
- **不要整段复制教材原文，要改写成学习笔记**
- **题目只能使用今天或之前已经讲过的知识点**
- **如果题目必须引入新概念，必须先在题目前补充讲解，再出题**
- **不要用后续章节的术语当例子，例如递归、哈希表、堆、图、动态规划等，除非当天已经讲到**

## 6. 每日讲义写法

阅读任务不要只写“阅读第几章”，还要把对应内容整理成讲义。

推荐格式：

```text
## 阅读任务

阅读书中：第 X 章 X.X-X.X

阅读时重点关注：
- **重点 1**
- **重点 2**
- **重点 3**

下面是对应内容的学习讲义版。这里不是复制教材原文，而是把教材内容改写成更适合学习和复习的笔记。
```

讲义写法：

- 按教材小节顺序整理
- 每个小节先讲核心概念
- 再给例子
- 最后给学习提醒
- 关键概念加粗

示例：

```text
### 1.2 标识符、对象和赋值语句

变量名不是盒子，而是对象的引用。

**对象有类型，变量名只是指向对象的标识符。**
```

## 7. 题目知识依赖检查

每日题目必须做一次“知识依赖检查”。检查目标是：**题目里不能突然出现前面没讲过的概念、语法、算法或工具**。

检查顺序：

1. 列出今天所有任务、练习、验收题里的关键词。
2. 判断这些关键词是否已经在今天或之前的讲义里解释过。
3. 如果没有解释过，优先替换成已学过的知识。
4. 如果这个概念当天必须使用，就把它前移到题目前，写成“做题前先理解”。
5. 如果概念属于后续章节，只做一句“后面会学”，不要把它作为今天题目的必要条件。

常见需要拦截的超前内容：

```text
递归
二分查找
哈希表
堆
树
图
动态规划
贪心
摊销分析
循环不变量
列表解析
生成器
类和继承
测试方法术语
```

示例：如果 Day 02 只讲了条件、循环、函数和异常，就不要在练习里要求使用列表解析，因为列表解析是 Day 03 的内容。

错误写法：

```python
def is_teen(age_data):
    teen_ages = [age for age in age_data if 13 <= age <= 19]
    return len(teen_ages) > 0
```

更合适的写法：

```python
def is_teen(age_data):
    for age in age_data:
        if 13 <= age <= 19:
            return True
    return False
```

示例：如果练习要求“边界测试”，但前文没有讲过边界测试，就必须先补一段：

```text
## 做练习前先理解：边界测试

边界测试就是专门测试输入范围边缘的数据，例如最小值、最大值，以及它们前后相邻的值。
```

然后再出题：

```text
给 `is_teen` 补边界测试：12、13、19、20。
```

原则：

- **先讲再练**
- **先用已学知识表达**
- **新概念要前置补充**
- **后续章节内容不要提前成为今天的任务要求**

## 8. 批量生成每日计划

样张确认后，再批量生成每日文件。

推荐目录：

```text
每日学习计划/
├── README.md
├── day01_第01周_第01天_主题.md
├── day02_第01周_第02天_主题.md
└── ...
```

每日学习计划目录必须有索引：

```text
# 每日学习计划索引

## 第 1 周：主题
- Day 01：[标题](day01_xxx.md)
- Day 02：[标题](day02_xxx.md)
```

原则：

- **文件名包含 day 编号、周编号、日编号、主题**
- **README 负责导航**
- **每个文件都是独立学习单元**

## 9. 质量检查

生成后检查：

- 文件数量是否正确
- README 链接是否存在
- Day 01 是否是确认过的精讲版
- 是否误放了不该推送的 PDF、图片或临时文件
- 是否存在重复文件
- 是否每个每日文件都有目标、讲义、任务、验收和复盘

建议检查命令：

```bash
find 每日学习计划 -maxdepth 1 -name 'day*.md' | wc -l
find . -type f -name '*.pdf'
git status -sb
```

额外做一次知识依赖检查：

```bash
rg -n "递归|二分查找|哈希|堆|树|图|动态规划|贪心|摊销|不变量|列表解析|生成器|边界测试" 每日学习计划/*.md
```

检查原则：

- 如果某个术语出现在题目里，确认它已经在当天题目前讲过。
- 如果只是作为“后面会学”的提醒，不要让它成为今天任务的一部分。
- 如果术语属于后续章节，优先删除或替换成当前已学内容。

## 10. 推送到 GitHub

如果本地还不是仓库，可以先克隆远程仓库：

```bash
git clone git@github.com:USER/REPO.git
```

复制学习计划文件：

```text
主题学习计划.md
主题每周学习计划.md
每日学习计划/
```

更新仓库 README，增加入口：

```text
## 系统学习计划

- [主题学习计划](主题学习计划.md)
- [主题每周学习计划](主题每周学习计划.md)
- [每日学习计划索引](每日学习计划/README.md)
```

提交并推送：

```bash
git add -A
git commit -m "Add topic study plans"
git push origin main
```

## 11. 复用提示词模板

以后学习其它主题时，可以这样发起：

```text
请按照当前文件夹里的资料，帮我制定一个系统学习计划。

要求：
1. 先读取资料目录和章节结构
2. 先做总学习计划
3. 再做每周学习计划
4. 先生成 Day 01 样张给我确认
5. 样张确认后，再批量生成每日独立 md 文件
6. 每日文件要包含阅读任务、改写讲义、重点加粗、实践任务、练习、验收和复盘
7. 不要大段复制原文，要改写成学习笔记
8. 每日题目不能使用今天之前没有讲过的知识点
9. 如果题目必须使用新概念，先在题目前补充讲解
10. 最后整理索引 README
```

如果要推送 GitHub，再补充：

```text
请把生成的学习计划推送到我的 GitHub 项目：
https://github.com/USER/REPO.git

只推送 Markdown 和必要代码，不要推送电子书原文件。
```

## 12. 输出文件清单

一次完整工作流结束后，理想产物是：

```text
主题学习计划.md
主题每周学习计划.md
每日学习计划/
├── README.md
├── day01_...
├── day02_...
└── dayN_...
学习计划生成工作流.md
```

## 13. 核心原则

- **先总后细**
- **先样张后批量**
- **先确认格式再扩展规模**
- **教材内容用改写讲义，不直接复制原文**
- **题目不能超前使用后续知识**
- **新概念必须先讲再练**
- **每天必须可执行**
- **每周必须可验收**
- **最终必须能沉淀到 GitHub 仓库**
